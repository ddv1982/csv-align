# csv-align Dependency Research — April 2026

Research compiled April 18, 2026. All versions reflect the latest stable
releases available on crates.io / npm at this date. Each section lists the
project's current pin, the latest stable version, notable migration concerns,
and idiomatic/modern guidance. Sources are linked inline.

---

## Rust Toolchain

- **Latest stable compiler:** `rustc 1.94.1` (released 26 Mar 2026). Only the
  latest stable channel is officially supported. — https://endoflife.date/rust
- **Recommended edition:** **Rust 2024 edition.** Edition 2024 was stabilized
  in **Rust 1.85.0** (Feb 20 2025) and is widely adopted a year later. New crates
  should opt in (`edition = "2024"` in `Cargo.toml`). Migrate with
  `cargo fix --edition`. — https://doc.rust-lang.org/edition-guide/rust-2024/
- **MSRV recommendation for this project:** **1.82+** is the practical floor
  because recent `axum 0.8.9` requires 1.80, Tauri 2.10 requires ~1.81, and
  `tracing` / `serde` / `tokio 1.50+` all want 1.75+. Pinning `rust-toolchain.toml`
  to `1.90` (LTS-like, Sept 2025) or newer is a reasonable choice.
- **Edition 2024 gotchas for this codebase:**
  - `if let` temporary scoping changed — existing `if let Some(x) = foo() { ... } else { ... }`
    chains may need `cargo fix --edition` rewrites.
  - `unsafe` requirements for setting env vars (`std::env::set_var`) — unlikely to
    affect csv-align, but test harnesses may flag.
  - Stricter lifetime capturing in RPIT (`impl Trait`) — mostly a non-issue for
    library code in this project.

## Rust Deps

Format: `current → latest` | migration notes | idiomatic suggestion.

### Web / HTTP stack

- **axum**: `0.7 → 0.8.9` (Apr 10 2026). **Breaking upgrade.**
  - Path params switched from `/:id` + `/*rest` to `/{id}` + `/{*rest}`
    (matchit 0.8 / OpenAPI-style).
  - `Option<T>` extractor now requires `OptionalFromRequestParts`/`OptionalFromRequest`
    impls; previously-silent errors now surface.
  - `#[async_trait]` removed from `FromRequest`/`FromRequestParts` (native async
    trait methods). Remove the attribute from custom extractors.
  - MSRV bumped to 1.80 (as of 0.8.9).
  - Sources: https://tokio.rs/blog/2025-01-01-announcing-axum-0-8-0
    https://github.com/tokio-rs/axum/releases
  - **Idiom:** prefer `Router::new().route("/users/{id}", get(handler))` and lean
    on native async fn extractors. Use `#[derive(FromRequestParts)]` from
    `axum-macros` where helpful.
- **tower**: `0.4 → 0.5.x`. `tower 0.5` was released in 2024 and is required
  transitively by `tower-http 0.6`. Main breaking change: `Service` trait's
  associated types and `Layer` composition stabilized; most app-level code only
  needs a bump. — https://github.com/tower-rs/tower/releases
- **tower-http**: `0.5 → 0.6.8` (Dec 8 2025). **Breaking upgrade.**
  - `body` module is gated behind specific features (`catch-panic`,
    `decompression-*`, `fs`, `limit`); enable what you need.
  - `TimeoutLayer::new()` deprecated in favor of `TimeoutLayer::with_status_code`.
  - Built against `tower 0.5`.
  - Source: https://github.com/tower-rs/tower-http/releases/tag/tower-http-0.6.0
- **tokio**: `1.x → 1.52.1` (LTS 1.51.x through Mar 2027). No breaking changes
  within 1.x; just bump. MSRV is 1.71. Prefer the `rt-multi-thread` + `macros`
  + `net` + `fs` feature set you actually need rather than `full`. —
  https://crates.io/crates/tokio

### Data / serialization

- **csv**: `1.3 → 1.3.x`. Still the de-facto Rust CSV library. No breaking API
  since 1.3.0 (Dec 2023); bug-fix releases only. Project was quiet in 2025/2026.
  - **Idiom:** prefer `ReaderBuilder::new().has_headers(true).flexible(true)`
    for user-supplied CSVs and `ByteRecord` for perf-critical paths.
- **encoding_rs_io**: `0.1 → 0.1.x`. Stable and low-activity; still the canonical
  bridge between `encoding_rs::Encoding` and `std::io::Read`. Keep as-is.
- **serde**: `1.0 → 1.0.x`. No migration needed. With edition 2024, consider
  `#[serde(deny_unknown_fields)]` on config structs for safer parsing.
- **serde_json**: `1.0 → 1.0.x`. No migration. For high-throughput pipelines,
  consider `sonic-rs` or `simd-json`, but `serde_json` remains the right default.

### Time / IDs / FS

- **chrono**: `0.4 → 0.4.x` (latest patch Feb 2026). Stable; no breaking
  change imminent. Alternative: `jiff` (2025 release by the `time`/chrono
  community) offers a cleaner timezone model — worth evaluating only if
  timezone logic grows. — https://crates.io/crates/chrono
- **uuid**: `1.0 → 1.18.1` (late Mar 2026). Seamless semver bump. Consider
  enabling `v7` feature if you want monotonic, time-ordered IDs (better DB
  locality than v4). — https://docs.rs/crate/uuid/latest
- **tempfile**: `3.8 → 3.14+`. API compatible; just bump. —
  https://crates.io/crates/tempfile

### Misc

- **strsim**: `0.10 → 0.11.x`. Minor API refinements only; bump is trivial.
  See "Stack-wide modernization picks" below for whether to keep it.
- **env_logger**: `0.10 → 0.11.x` (latest release in 2026). API compatible.
  See "Stack-wide modernization picks" — strongly consider migrating to
  `tracing` + `tracing-subscriber`.
- **log**: `0.4 → 0.4.x`. Keep if staying with `env_logger`; `tracing` has a
  `tracing-log` shim that ingests `log` records, so no disruption if you
  migrate.

## React 19 Migration

### Key changes

1. **`ref` as a prop.** Function components can take `ref` directly in props;
   `forwardRef` is on a deprecation path.
   ```tsx
   function MyInput({ placeholder, ref }: { placeholder: string; ref?: React.Ref<HTMLInputElement> }) {
     return <input placeholder={placeholder} ref={ref} />
   }
   ```
2. **`<Context>` as provider** instead of `<Context.Provider>`. Old form still
   works but is on a deprecation path.
3. **Actions + `useActionState` + `useOptimistic` + `useFormStatus`** —
   React-managed form submission / pending state / optimistic updates.
4. **`use(promise)` / `use(context)`** — first-class way to unwrap promises in
   render (suspends) and conditionally read context.
5. **Native document metadata** — `<title>`, `<meta>`, `<link>` in any component
   are hoisted to `<head>`. Lets you delete `react-helmet`.
6. **Built-in stylesheet / async script / preload support** (`preinit`,
   `preload`, `prefetchDNS`, `preconnect` from `react-dom`).
7. **Ref cleanup functions** — `ref={el => { ...; return () => { ...cleanup } }}`.
   TypeScript now rejects implicit returns from ref callbacks.
8. **Improved hydration error reporting** with diffs.
9. **Custom elements** — full support, props map to properties on the client.
10. **React DOM static APIs** — `prerender`, `prerenderToNodeStream`.

Source: https://react.dev/blog/2024/12/05/react-19

### Codemods

- Codemods live in the community-maintained `react-codemod` repo; preferred
  runner is `codemod` (the `codemod` CLI, ex Codemod.com). Highlights:
  - `replace-reactdom-render`
  - `replace-use-form-state` (rename to `useActionState`)
  - `remove-forward-ref`
  - `context-provider-to-context`
- **Types codemod:** `npx types-react-codemod preset-19 ./src`
  (repo: https://github.com/eps1lon/types-react-codemod). Includes
  `no-implicit-ref-callback-return`, `deprecated-react-type-arguments`, etc.
- Guide: https://react.dev/blog/2024/04/25/react-19-upgrade-guide
  and https://docs.codemod.com/guides/migrations/react-18-19

### `@types/react` 19 updates required

- Drop from 18.2 → 19.x. Main surface changes:
  - `ReactElement`, `JSX.Element`, etc. are stricter — some `any` holes now
    surface as errors. The codemod catches most of them.
  - Ref callback return types now matter (see #7 above).
  - `ReactNode` no longer implicitly allows bigint in a couple of edge cases.
- If you use `@types/react-dom`, bump together.

### React Compiler (optional)

- React Compiler reached beta Oct 2024; it auto-memoizes components so you can
  drop most `useMemo`/`useCallback`/`React.memo` calls.
- To enable with Vite 8: `@vitejs/plugin-react v6` exposes
  `reactCompilerPreset`, opt-in via `@rolldown/plugin-babel`. Not required but
  a reasonable performance win for this app.
- Docs: https://react.dev/learn/react-compiler

## Stack-Wide Modernization Picks

### Logging: env_logger → tracing? — **Yes, migrate.**

- `tracing` + `tracing-subscriber` is the de-facto standard in 2026. It covers
  structured logging, spans (async-aware), and is the format `axum`, `tower`,
  `tokio`, and `tauri` emit into natively.
- For a drop-in replacement behavior identical to env_logger, use:
  ```rust
  tracing_subscriber::fmt()
      .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
      .init();
  ```
  This respects `RUST_LOG` just like `env_logger`.
- Dependencies:
  - `tracing = "0.1"`
  - `tracing-subscriber = { version = "0.3", features = ["env-filter", "fmt"] }`
  - Optional: `tracing-log = "0.2"` to capture any transitive `log::*` calls
    from libraries.
- **Migration effort:** low. `log::info!` → `tracing::info!`; everything else
  compiles unchanged when `tracing-log` is initialized. Then delete `log`
  and `env_logger` from `Cargo.toml`.
- References:
  - https://docs.rs/tracing
  - https://blog.logrocket.com/comparing-logging-tracing-rust/

### Error handling: anyhow / thiserror — **Adopt both.**

- **`thiserror`** for library-layer enums in `src/` (typed, structured,
  preserves `?` ergonomics, plays well with axum's `IntoResponse`).
- **`anyhow`** for application boundaries (`src/main.rs`, `src-tauri` command
  handlers) where you only need to bubble context to the user/log.
- Typical pairing:
  ```rust
  // src/errors.rs
  #[derive(thiserror::Error, Debug)]
  pub enum AlignError {
      #[error("csv parse error: {0}")] Csv(#[from] csv::Error),
      #[error("io error: {0}")]       Io(#[from] std::io::Error),
      // ...
  }
  // axum:
  impl IntoResponse for AlignError { /* map to JSON + status */ }
  ```
- References:
  - https://github.com/dtolnay/anyhow
  - https://docs.rs/thiserror

### Replace `strsim`? — **Optional; stay with it unless there's a concrete pain.**

- `strsim` is feature-frozen but correct and cheap (pure algorithmic Levenshtein,
  Jaro, etc.). Good for header-matching / column-guessing use cases.
- Alternatives worth knowing, but only adopt if the use case warrants:
  - **`rapidfuzz` (Rust port)** — faster at scale, richer API, familiar to
    Python users. crates.io: `rapidfuzz`. Use if doing large N×M header scoring.
  - **`nucleo`** — fast **fuzzy-finder** matcher from the Helix editor team.
    Interactive-search-style (sublime/fzf ranking). Wrong tool for "which of
    these two header names is more similar"; right tool for "user types and
    we narrow options."
  - **`code-fuzzy-match`** — VS Code-style fuzzy matching.
- Recommendation: **keep `strsim`** and bump to `0.11`.

### Linter for the frontend: **Add `oxlint`.**

- csv-align's `frontend/` has no JS/TS linter configured (only TypeScript +
  Vitest). In 2026 the pragmatic picks are:
  - **ESLint v9** (flat config) — most coverage, most plugins, slowest.
  - **Biome** — fast all-in-one (lint + format), moderate rule coverage,
    single config.
  - **oxlint** — fastest (10–100× ESLint), zero-config, subset of ESLint
    rules, built on Oxc (same stack as Rolldown/Vite 8).
- For a project of this size, **oxlint** gives 90%+ of the value with zero
  config friction: `npm i -D oxlint` then `npx oxlint src`. It pairs well
  with TypeScript's type-checking to catch most defects.
- If you later need JSX-a11y or framework rules oxlint lacks, layer Biome
  or ESLint on top — they coexist.
- References:
  - https://www.pkgpulse.com/blog/oxc-vs-eslint-vs-biome-javascript-linting-2026
  - https://oxc.rs/docs/guide/usage/linter.html

### Misc stack-wide notes

- **Rust edition 2024** is worth adopting alongside the axum 0.8 bump — one
  coordinated PR with `cargo fix --edition`, clippy fixes, and axum path-syntax
  changes.
- **Cargo.toml workspace MSRV** — consider adding `rust-version = "1.82"`
  (or newer) to both crates to make dependency compatibility explicit and
  give clippy a stable target.
- **`cargo audit`** in CI will flag the transitive `atk`/`gtk-sys`/`fxhash`
  unmaintained warnings from wry/tao regardless of what we do; the Tauri team
  has acknowledged them and is tracking a `wry` upgrade to GTK4. Safe to
  allow-list until Tauri 3.

## Summary Recommendations (priority order)

1. **Axum 0.7 → 0.8 + tower-http 0.5 → 0.6 + tower 0.5** — bundled breaking
   bump, coordinated with a Rust 2024 edition migration.
2. **React 18.2 → 19** using the `types-react-codemod preset-19` + React
   codemods. Update `@types/react` / `@types/react-dom` at the same time.
3. **env_logger → tracing + tracing-subscriber.** Swap in `RUST_LOG`
   compatibility, delete `env_logger` + `log`.
4. **Add `anyhow` + `thiserror`** and replace ad hoc `Box<dyn Error>` /
   `String` errors.
5. **Bump Tauri pins to 2.10.x** (+ plugin-dialog 2.10.x). No source changes.
6. **Pin to TypeScript 5.8** first; evaluate 6.0 once the `types: []` default
   is mapped out.
7. **Add `oxlint` to the frontend** as a CI step (`npm run lint`).
8. **Patch-bump** everything else: `uuid → 1.18`, `tokio → 1.52`, `tempfile → 3.14`,
   `strsim → 0.11`, `@testing-library/react → 16.3.2`, etc.
